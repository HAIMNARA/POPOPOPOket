# 10 - 최적 워크플로우 (The Best Approach)

## 핵심 발견 요약

### Grokemon에서 배운 것:
- InputGate를 통한 3단계 입력 안전 장치는 시스템 안정성의 핵심입니다.
- 모드별로 도구를 필터링하여 잘못된 액션이 발생하는 상황을 원천 차단합니다.
- A* 경로탐색 알고리즘과 NPC 장애물 감지 기능을 결합하여 효율적인 이동을 구현했습니다.
- 액션을 실행한 뒤 인터럽트를 발생시키고, 상태를 갱신한 후 다음 액션을 취하는 상태 동기화가 필수적입니다.
- 6개 섹션(objectives, journal, notes, strategy, landmarks, lessons)으로 구성된 에이전트 메모리를 통해 장기 컨텍스트를 유지합니다.
- Supervisor가 5개 레벨의 막힘 현상(actionLoop, actionCycle, locationLoop, noProgress, backtrackLoop)을 감지합니다.
- 매 턴마다 JSON 형식으로 증거를 기록하는 것은 디버깅의 생명줄 역할을 합니다.
- Grok 4.3 모델이 비전 정보와 게임 컨텍스트를 처리할 때 가장 뛰어난 성능을 보여주었습니다.

### POPOPOPOket에서 배운 것:
- 메모리 마스킹(green/red) 기법을 도입하여 벽 탐색에 수십 턴이 걸리던 과정을 1에서 2턴으로 단축하는 혁명적인 성과를 거두었습니다.
- 3단계 마스킹 진화(green: 빈 공간, yellow: 풀숲, red: 벽)를 통해 풀숲을 벽으로 오인식하는 문제를 해결했습니다.
- fast-flow 설계를 적용해 Route 1 같은 직선 구간은 LLM 없이 자동으로 통과하여 토큰을 70% 이상 절약했습니다.
- stuck-memory 기법을 통해 동일 좌표에서 8회 반복 실패할 경우 해당 정보를 프롬프트에 주입하여 무한 반복을 차단합니다.
- 비전 입력의 오염을 방지하기 위해 원본 화면 이미지와 가이드 이미지를 항상 분리하여 제공해야 합니다.
- 닫힌 피드백 루프는 시스템을 지속적으로 개선하기 위한 필수 조건입니다.
- 세션 핸드오프 프로토콜을 정의하여 Markdown 문서를 정규화하고 압축 및 정리합니다.
- 에뮬레이터 실행 시 `--script` CLI 옵션을 사용하면 안 되며, Lua 스크립트는 GUI 로드 시 1회만 실행해야 중복 로드로 인한 크래시를 막을 수 있습니다.

### PokemonRedExperiments에서 배운 것:
- 좌표 기반 탐색 보상 방식이 KNN 방식보다 4배 빠르고 단순합니다.
- 이벤트 플래그(event flags)에 4를 곱하고, 획득한 배지(badges)에 10을 곱하며, 체력 회복(healing)에 10을 곱한 보상 조합이 효과적입니다.
- 64개의 병렬 환경을 동시에 구동하여 훈련 속도를 극대화했습니다.
- 화면 이미지, 체력, 레벨, 배지, 이벤트, 맵 정보를 포함하는 딕셔너리(Dict) 관측 공간이 MultiInputPolicy에 가장 적합합니다.
- 플레이어 레벨을 입력할 때 스칼라 값 대신 Fourier 인코딩을 적용하여 더 풍부한 표현을 제공합니다.
- 동일 위치를 600회 이상 방문할 때 stuck penalty(-0.05)를 부여하여 탐색 정체를 방지합니다.
- WebSocket 스트리밍을 구현하여 여러 에이전트의 탐색 상태를 실시간으로 모니터링할 수 있습니다.

### 외부 연구에서 배운 것:
- LLM이 상위 계획을 세우고 RL 또는 결정론적 알고리즘이 하위 실행을 담당하는 하이브리드 접근법이 순수 LLM이나 RL 단독 모델보다 우수합니다.
- BALROG 벤치마크에 따르면 LLM은 보상 모델로서 탁월한 성능을 보이지만, 직접 게임을 조종하는 에이전트로서는 한계가 있습니다.
- Continual Harness 프로젝트는 온라인 적응을 통해 하네스를 자동으로 개선하는 자기개선 에이전트를 제안했습니다.
- 사전훈련 후 지도 미세조정(SFT)을 거쳐 강화학습(RL)을 진행하는 다단계 훈련 방식이 단독 훈련보다 효과적입니다.
- MCP 서버를 구축하여 여러 에뮬레이터 인스턴스를 동시에 운영할 수 있습니다.

## 최적 워크플로우 제안

### Phase 1: 환경 선택

```
에뮬레이터 선택 결정 트리:

Q1: 프레임 단위의 정밀한 제어가 필요한가?
  |
  +-- 예: PyBoy 사용 (Python 네이티브 환경, 최대 344배속 실행 가능)
  |
  +-- 아니오:
        |
        Q2: 기존에 구축된 mGBA 인프라가 존재하는가?
              |
              +-- 예: mGBA + mGBA-http 조합 사용 (HTTP API 기반 제어)
              |
              +-- 아니오: PyBoy 사용 (설치가 단순하며 의존성이 최소화됨)
```

에뮬레이터 환경은 프로젝트의 요구사항에 따라 선택합니다. 프레임 단위의 정밀한 속도 제어와 빠른 학습이 필요하다면 PyBoy를 선택하는 것이 좋습니다. 기존 mGBA 인프라가 이미 구축되어 있다면 mGBA-http를 통해 HTTP API로 제어하는 방식을 권장합니다.

### Phase 2: 에이전트 아키텍처

```
목표 범위 결정 트리:

Q1: 게임을 어디까지 진행할 것인가?
  |
  +-- Viridian City까지:
  |     단일 LLM 에이전트 + fast-flow + stuck-memory 조합
  |     (Grokemon 방식 적용, 최소 7시간 소요 예상)
  |
  |-- 체육관 배지 획득까지:
  |     하이브리드 아키텍처: LLM 상위 계획 + 결정론적 하위 실행
  |     + 메모리 마스킹 + 세션 핸드오프 프로토콜 적용
  |
  +-- 전체 클리어까지:
        다층 아키텍처 (L1 무의식 / L2 비평가 / L3 정책 LLM)
        + 다중 인스턴스 병렬 탐색 환경 구축
        + 중앙 전략 가이드 자동 업데이트 시스템
        + 코딩 에이전트를 통한 하네스 자동 개선 루프
```

목표로 하는 게임 진행 범위에 따라 에이전트 아키텍처를 다르게 설계해야 합니다. 단순한 초반 구역 도달이 목표라면 단일 LLM 에이전트로 충분합니다. 체육관 배지 획득이나 전체 클리어를 목표로 한다면 상위 계획과 하위 실행을 분리하는 하이브리드 또는 다층 아키텍처를 도입해야 합니다.

### Phase 3: 핵심 컴포넌트 구현 순서

성공적인 시스템 구축을 위해 아래 순서대로 컴포넌트를 단계별로 구현합니다.

1. **에뮬레이터 연결** (Day 1)
   - PyBoy 또는 mGBA-http 환경을 설정합니다.
   - RAM 데이터를 읽고 버튼 입력을 검증하는 기본 API를 작성합니다.
   - 화면 스크린샷을 캡처하는 파이프라인을 구축합니다.

2. **게임 상태 읽기** (Day 1-2)
   - 플레이어의 현재 맵 ID와 X, Y 좌표(0xD35E, 0xD361, 0xD362)를 읽어옵니다.
   - 배틀 상태(0xD057), 파티 정보(0xD163), 획득한 배지(0xD356)를 파싱합니다.
   - 입력 안전을 판단하기 위해 wJoyIgnore, wWalkCounter, rWY 메모리 주소를 모니터링합니다.

3. **입력 안전 게이트** (Day 2)
   - wJoyIgnore 값이 0인지 검사합니다.
   - wWalkCounter 값이 0인지 확인하여 플레이어가 이동 중이 아닌 상태를 보장합니다.
   - 텍스트 윈도우가 열려 있는지 상태를 검사합니다.
   - 2회 연속으로 동일한 RAM 상태가 읽힐 때까지 폴링하여 데이터의 안정성을 확보합니다.

4. **모드 분류** (Day 2-3)
   - 현재 게임 화면을 title, overworld, battle, dialog, naming 모드로 분류합니다.
   - 각 모드에 따라 허용되는 액션 도구를 필터링하여 오작동을 방지합니다.

5. **LLM 관측 빌더** (Day 3)
   - 원본 스크린샷과 마스킹 처리된 이미지를 분리하여 준비합니다.
   - RAM에서 추출한 플레이어 상태 정보를 텍스트 요약으로 변환합니다.
   - 최근 실행한 10개의 명령 히스토리를 포함합니다.
   - stuck-memory 정보를 프롬프트에 주입합니다.
   - Supervisor가 제공하는 힌트를 관측 데이터에 추가합니다.

6. **fast-flow 자동조종** (Day 3-4)
   - Route 1과 같이 단순한 직선 경로는 LLM 호출 없이 자동으로 북진하도록 코드를 작성합니다.
   - 배틀 모드 진입 시 최대 40턴 동안 자동으로 A 버튼을 누르도록 제어합니다.
   - 대화창이 나타나면 자동으로 A 버튼을 눌러 진행합니다.
   - 자동조종이 실패하거나 막히면 즉시 LLM 에이전트로 제어권을 넘깁니다.

7. **stuck-memory** (Day 4)
   - 플레이어의 좌표와 시도한 액션 키를 조합하여 실패 횟수를 카운팅합니다.
   - 동일한 상황에서 8회 이상 실패하면 해당 좌표와 실패 원인을 프롬프트에 강제로 주입합니다.
   - 메모리 효율을 위해 최대 6개의 실패 엣지만 보존하도록 관리합니다.

8. **메모리 마스킹** (Day 4-5)
   - 에뮬레이터의 충돌(collision) 데이터를 파싱합니다.
   - 통과 가능한 빈 공간(green), 풀숲(yellow), 벽(red)의 3단계로 맵을 마스킹합니다.
   - LLM의 비전 입력으로 원본 이미지와 마스킹 이미지를 함께 제공하여 경로 탐색 효율을 높입니다.

9. **증거 기록** (Day 5)
   - 매 턴마다 게임 상태, 프롬프트, 모델 응답, 도구 호출 로그를 JSON 파일로 저장합니다.
   - 액션 실행 전후의 스크린샷을 함께 기록합니다.
   - 기록 저장 시 API 키나 개인정보 등 비밀 정보가 노출되지 않도록 삭제 처리합니다.

10. **Supervisor** (Day 5-6)
    - 5개 레벨의 막힘 감지 알고리즘을 구현하여 에이전트의 정체를 감시합니다.
    - 최종 목표와 세부 목표를 관리하는 목표 원장(Goal Ledger)을 운영합니다.
    - 에이전트가 무한 루프에 빠지면 최대 20개의 입력을 직접 주입하는 개입 루프를 실행합니다.

### Phase 4: 최적 모델 선택

| 모델 | 강점 | 약점 | 추천 용도 |
| :--- | :--- | :--- | :--- |
| Grok 4.3 | 비전 정확도가 높고 게임 컨텍스트 이해력이 뛰어나며 토큰 효율이 좋음 | 프록시 설정이 필요함 | 기본 에이전트 모델 |
| GPT-5.4 | 운영 비용이 저렴함 | 맵 탈출 시 매핑 오류가 발생할 수 있음 | 비용 최적화가 필요할 때 |
| Claude | 코딩 및 테스트 하네스 설계 능력이 우수함 | 비전을 통한 게임 제어 성능이 다소 약함 | 하네스 및 도구 개발 |
| Gemini 3 Pro | 비용 대비 완주 성공률이 가장 균형 잡힘 (Pareto 최적) | 특별한 약점 없음 | 장기 실행 및 대규모 탐색 |

각 AI 모델은 고유한 특성을 가지고 있습니다. 프로젝트의 예산과 목표 구역에 맞춰 적절한 모델을 선택해야 합니다. 기본 에이전트로는 Grok 4.3을 추천하며, 장기 실행 시에는 Gemini 3 Pro가 가장 합리적인 선택입니다.

### Phase 5: 운영 체크리스트

- [ ] 에뮬레이터와 ROM 파일이 정상적으로 실행되는지 확인합니다.
- [ ] RAM에서 맵 ID, 플레이어 좌표, 배틀 상태를 정확히 읽어오는지 검증합니다.
- [ ] 입력 안전 게이트가 대기 상태와 이동 상태를 올바르게 판별하는지 테스트합니다.
- [ ] Route 1 구간에서 fast-flow 자동조종이 LLM 없이 작동하는지 확인합니다.
- [ ] stuck-memory 기법이 작동하여 실패 좌표 정보가 프롬프트에 주입되는지 검증합니다.
- [ ] 3단계 메모리 마스킹 이미지가 올바르게 생성되는지 확인합니다.
- [ ] LLM API 연결 상태와 관측 빌더의 데이터 포맷을 점검합니다.
- [ ] 매 턴의 실행 결과가 JSON 파일과 스크린샷으로 증거 디렉토리에 기록되는지 확인합니다.
- [ ] 세션 핸드오프 프로토콜에 따라 Markdown 문서가 정상적으로 정리되는지 검토합니다.

## 반드시 피해야 할 것 (Anti-Patterns)

시스템 구축 및 운영 시 실패를 방지하기 위해 다음 사항들을 반드시 피해야 합니다.

1. **에뮬레이터 실행 시 `--script` CLI 옵션 사용 금지**
   - mGBA 실행 시 이 옵션을 사용하면 에뮬레이터가 즉시 종료되는 문제가 발생합니다.

2. **Lua 스크립트 중복 로드 금지**
   - Lua 스크립트는 GUI 로드 시 1회만 실행해야 하며, 중복으로 로드하면 에뮬레이터가 크래시됩니다.

3. **원본 이미지에 좌표 그리드 직접 오버레이 금지**
   - 화면 정보가 오염되면 LLM의 비전 인식 성능이 크게 저하되므로 원본과 가이드 이미지는 항상 분리해야 합니다.

4. **이진 마스킹 사용 금지**
   - 풀숲과 벽을 구분하지 못하는 이진 마스킹은 에이전트가 풀숲을 벽으로 오인하게 만드므로 반드시 3단계 마스킹을 적용해야 합니다.

5. **토큰 절감만을 목적으로 한 무리한 최적화 금지**
   - 비용을 줄이기 위해 필요한 컨텍스트를 과도하게 생략하면 진행도가 저하되므로 성능과 비용의 균형을 맞춰야 합니다.

6. **증거 없는 개선 시도 금지**
   - 턴별 JSON 기록과 스크린샷 등 명확한 데이터 증거 없이 감에 의존하여 프롬프트나 코드를 수정하면 안 됩니다.

7. **ROM 정체성 혼동 금지**
