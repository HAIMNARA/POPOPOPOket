# 입력 안전 메커니즘 (Input Safety)

포켓몬스터 적/녹(Pokemon Red/Green)과 같은 8비트 Game Boy 환경에서 에이전트가 게임을 안정적으로 조작하기 위해서는 정교한 입력 안전 메커니즘이 필수적입니다. 게임 엔진의 내부 상태를 고려하지 않고 무작위로 입력을 주입하면 입력이 무시되거나, 원치 않는 메뉴가 열리거나, 심각한 경우 게임 상태가 손상될 수 있습니다. 본 문서에서는 Grokemon, PokemonRedExperiments, 그리고 POPOPOPOket 프로젝트에서 구현된 입력 안전 메커니즘의 설계와 동작 방식을 상세히 분석합니다.

---

## 1. Grokemon InputGate (input-gate.ts)

Grokemon 프로젝트의 `input-gate.ts` 파일은 약 248줄의 코드로 구성되어 있으며, 에뮬레이터에 입력을 전달하기 직전에 게임의 내부 메모리 상태를 검사하여 입력의 유효성을 검증하는 핵심 게이트웨이 역할을 수행합니다.

### 3단계 검증 프로세스 (Three-Step Verification)

InputGate는 입력을 주입하기 전에 Game Boy의 WRAM(Work RAM) 영역에서 세 가지 핵심 주소를 읽어 들여 검증을 수행합니다.

1. **wJoyIgnore (0xCDCB) 검사**
   - 메모리 주소 `0xCDCB` 값이 `0`이 아닌 경우 입력을 즉시 차단합니다.
   - 이 상태는 게임 스크립트나 컷신이 실행 중이어서 시스템이 플레이어의 입력을 의도적으로 잠근 상태를 의미합니다. 차단 시 반환되는 오류 코드는 `"joy-ignore"`입니다.

2. **wWalkCounter (0xD00D) 검사**
   - 메모리 주소 `0xD00D` 값이 `0`이 아닌 경우 입력을 차단합니다.
   - 캐릭터가 한 타일에서 다른 타일로 이동하는 걷기 애니메이션이 진행 중임을 나타냅니다. 이 타이밍에 새로운 입력이 들어가면 프레임 드롭이나 입력 누락이 발생할 수 있습니다. 차단 시 반환되는 오류 코드는 `"walk-animation"`입니다.

3. **rWY (0xFF4A) 검사**
   - 메모리 주소 `0xFF4A`는 화면의 Window Y 좌표를 나타냅니다. 이 값이 `144`보다 작으면 화면 하단에 대화창이나 메뉴 창이 활성화되어 있음을 의미합니다.
   - 기본적으로 이 상태에서는 방향키 입력을 차단하여 대화가 끊기거나 메뉴가 오작동하는 것을 방지합니다. 차단 시 반환되는 오류 코드는 `"text-window"`입니다.
   - 예외 처리: 에이전트가 명시적으로 대화창을 넘기기 위해 A, B, Start 버튼을 입력하거나, 시스템 설정에서 `allowDialog` 플래그가 활성화된 경우에는 입력을 허용합니다.

### Settle 메커니즘 (Settle Mechanism)

입력이 주입된 후 게임 화면과 메모리가 완전히 안정화될 때까지 대기하는 메커니즘입니다.

- **동작 방식**: 50ms 간격으로 게임 상태를 폴링(Polling)합니다.
- **안정 조건**: 2회 연속으로 동일하고 안정적인 메모리 값을 읽어올 때까지 대기합니다.
- **타임아웃**: 최대 5초 동안 대기하며, 이 시간 내에 안정 상태에 도달하지 못하면 타임아웃을 발생시키고 다음 단계로 진행합니다.

## 2. PokemonRedExperiments (PyBoy)

PokemonRedExperiments 프로젝트는 PyBoy 에뮬레이터 환경을 기반으로 강화학습을 진행하는 프레임워크입니다. Grokemon과 비교했을 때 입력 안전 메커니즘의 설계 철학이 크게 다릅니다.

- **안전 메커니즘의 부재**: 메모리 상태를 실시간으로 검사하여 입력을 차단하는 동적 게이트웨이가 존재하지 않습니다.
- **고정 액션 주기 (Fixed Action Cycle)**: 에이전트는 고정된 프레임 주기에 맞춰 입력을 주입합니다.
  - **프레스(Press)**: 0프레임에 버튼을 누릅니다.
  - **릴리스(Release)**: 8프레임에 버튼을 뗍니다.
  - **틱(Tick)**: 총 24프레임 동안 게임을 진행(Step)시킵니다.
- **입력 무시 현상**: 에이전트가 게임 엔진이 입력을 받을 수 없는 안전하지 않은 타이밍(예: 대화창 애니메이션 중)에 입력을 주입하더라도, 시스템은 이를 차단하지 않고 그대로 에뮬레이터에 전달합니다. 결과적으로 게임 엔진 수준에서 해당 입력이 무시되며, 에이전트는 스스로 이러한 무시 현상을 학습해야 합니다.

## 3. POPOPOPOket Supervisor 정규화 (Normalization)

POPOPOPOket의 Supervisor 모듈은 에이전트의 고수준 명령을 Game Boy가 이해할 수 있는 안전하고 정규화된 입력 스트림으로 변환합니다.

### 방향 이동 1타일 정규화

- **SINGLE_TILE_DURATION**: 캐릭터가 1타일을 이동하는 데 필요한 표준 시간은 12프레임으로 정의됩니다.
- Supervisor는 에이전트가 방향키를 입력할 때 정확히 12프레임 동안만 버튼이 눌린 상태를 유지하도록 강제하여 오버슈팅(Overshooting)이나 언더슈팅(Undershooting)을 방지합니다.

### 비방향 탭 정규화

- A, B, Start, Select와 같은 비방향성 버튼 입력은 고정된 최소 프레임 동안만 눌렸다가 떼어지도록 정규화됩니다. 이를 통해 연속 입력으로 인한 대화창 더블 클릭 현상을 방지합니다.

### 위험한 방향 다중홀드 거부

- Up과 Down, 또는 Left와 Right처럼 물리적으로 불가능하거나 게임 엔진의 오작동을 유발할 수 있는 다중 방향키 동시 입력을 감지하고 거부합니다.

### 정착 프레임 대기 및 폴링

- 입력이 완료된 후 캐릭터가 타일 그리드에 완전히 정착할 때까지 대기 프레임을 삽입합니다.
- 화면이 검게 변하는 맵 전환(Loading) 프레임이 감지되면, 화면이 다시 정상적으로 렌더링되고 관측이 가능해질 때까지 폴링을 수행하며 대기합니다.

## 4. Grokemon Guards (Guards.ts)

`Guards.ts` 파일은 에이전트가 게임의 규칙을 위반하거나 비정상적인 상태에 빠지는 것을 방지하기 위한 정적 검증 규칙들을 정의합니다.

### 배틀 명령 검증 (Battle Command Validation)

전투 상황에서 에이전트가 선택한 행동이 유효한지 검증합니다.

- **기술 존재 여부**: 선택한 기술 슬롯에 실제 기술이 존재하는지 확인합니다.
- **PP 검사**: 해당 기술의 남은 PP(Power Points)가 0보다 큰지 확인합니다. PP가 0인 기술은 선택할 수 없습니다.
- **아이템 보유 검사**: 도구 창에서 아이템을 사용하려 할 때, 해당 아이템의 개수가 1개 이상 존재하는지 확인합니다.
- **포켓몬 HP 검사**: 교체하려는 포켓몬의 HP가 0보다 큰지 확인합니다. 기절한 포켓몬으로의 교체 시도는 차단됩니다.

### 네비게이션 검증 (Navigation Validation)

- 에이전트가 이동하려는 목표 좌표가 현재 맵의 유효한 경계 범위 내에 존재하는지 확인합니다. 맵 밖으로 벗어나려는 비정상적인 이동 명령은 사전에 차단됩니다.

### 트레이너 배틀 도주 불가 규칙

- 야생 포켓몬과의 전투와 달리, 일반 트레이너나 체육관 관장과의 배틀에서는 도망치기(Run) 명령이 불가능합니다. Guards는 트레이너 배틀 상태를 감지하여 도망치기 입력을 원천 차단하고 에이전트가 전투에 집중하도록 강제합니다.

## 5. InputReadiness 폴링 (InputReadiness.ts)

`InputReadiness.ts`는 게임의 입력 준비 상태를 실시간으로 모니터링하고 진단하는 모듈입니다.

### 동시 읽기 메커니즘

- 효율성을 극대화하기 위해 `wJoyIgnore`, `wWalkCounter`, `rWY` 세 가지 핵심 메모리 주소를 단일 배치 읽기 요청으로 동시에 가져옵니다.

### ReadinessState 구조

시스템은 다음과 같은 구조체로 입력 준비 상태를 관리합니다.

```typescript
interface ReadinessState {
  overworldReady: boolean;
  lockReasons: string[];
}
```

- `overworldReady`: 캐릭터가 필드에서 자유롭게 움직일 수 있는 상태인지 여부를 나타내는 불리언 값입니다.
- `lockReasons`: 입력을 방해하는 요인들이 배열 형태로 저장됩니다. 예컨대 `["joy-ignore", "walk-animation"]`과 같이 구체적인 원인을 기록하여 디버깅을 용이하게 합니다.

### MiniState 모드 분류 및 적용

- 시스템은 현재 게임의 세부 상태를 나타내는 MiniState를 분석합니다.
- 현재 상태가 오버월드(Overworld), 배틀(Battle), 대화창(Dialog), 메뉴(Menu) 중 어디에 해당하는지 분류하고, 각 모드에 최적화된 입력 대기 시간과 검증 규칙을 동적으로 적용합니다.
