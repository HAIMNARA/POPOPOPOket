# 미래 로드맵 (Future Roadmap)

포켓몬스터 자율 플레이 에이전트 프레임워크의 지속적인 발전과 성능 향상을 위한 단계별 개발 로드맵입니다. 본 문서는 단기적인 시스템 안정화 과제부터 장기적인 아키텍처 진화 방향, 그리고 학계 및 오픈소스 프로젝트의 최신 동향을 반영한 최종 비전을 제시합니다.

---

## 1. P0: 즉시 개선 과제 (Immediate Improvements)

가장 우선순위가 높으며 현재 시스템의 병목 현상을 해결하기 위해 즉각적으로 반영해야 할 개선 사항들입니다.

### 컨텍스트 압축 및 세션 핸드오프 프로토콜
- **목적**: LLM의 컨텍스트 윈도우 한계로 인한 토큰 비용 상승과 추론 속도 저하 문제를 해결합니다.
- **방안**: 이전 턴의 세부 실행 이력을 요약(Summarization)하여 핵심 상태 정보만 남기는 압축 알고리즘을 도입합니다. 세션이 전환되거나 에뮬레이터가 재시작될 때, 이전 세션의 요약된 메모리와 맵 탐색 그래프를 끊김 없이 전달하는 핸드오프(Handoff) 프로토콜을 설계합니다.

### 풀숲 3단계 마스킹 (green/yellow/red)
- **목적**: 에이전트가 풀숲 타일을 벽으로 오인하여 우회하거나, 반대로 위험을 인지하지 못하고 무방비하게 진입하는 문제를 해결합니다.
- **방안**: 맵 타일 파싱 단계에서 풀숲 속성을 가진 타일을 식별하여 노란색(yellow)으로 마스킹합니다. 이를 통해 에이전트가 이동 가능성(Passability)과 전투 발생 확률(Battle Probability)을 동시에 인지하도록 돕습니다.

### 침실 스폰 가이드 프롬프트 주입
- **목적**: 게임 시작 직후 태초마을 침실에서 아래층으로 내려가는 계단을 찾지 못해 발생하는 초기 정체 현상을 방지합니다.
- **방안**: 게임 시작 세션의 첫 5턴 동안 "현재 위치는 2층 침실이며, 아래층으로 내려가기 위해 화면 우측 하단의 계단 타일로 이동해야 한다"는 명시적인 힌트 프롬프트를 주입합니다.

### pre-harness 포트 8888 검증 가드
- **목적**: 에뮬레이터와 통신하는 HTTP 서버가 포트 충돌이나 미종료 프로세스로 인해 실행되지 않는 문제를 방지합니다.
- **방안**: 하네스 시스템이 구동되기 전에 포트 8888의 사용 여부를 확인하고, 이미 사용 중인 경우 기존 프로세스를 안전하게 종료하거나 빈 포트를 동적으로 할당하는 검증 가드 스크립트를 추가합니다.

## 2. P1: 차기 마일스톤 및 이벤트 목표 (Next Event Goals)

시스템의 구조적 한계를 극복하고 게임 플레이의 완성도를 높이기 위한 차기 개발 단계입니다.

### PyBoy 프레임워크 도입
- **목적**: 기존의 `mGBA + Lua 스크립트 + HTTP 통신`으로 이어지는 복잡한 툴체인을 단순화하여 실행 속도와 안정성을 극복합니다.
- **방안**: Python 기반의 고속 에뮬레이터인 PyBoy를 도입합니다. 이를 통해 별도의 소켓 서버나 HTTP 통신 없이 단일 Python 프로세스 내에서 직접 메모리를 읽고 입력을 주입하는 하이브리드(Python/TypeScript) 아키텍처로 전환합니다.

### 다층 아키텍처 표준화 (L1/L2/L3)
- **목적**: 에이전트의 의사결정 단계를 계층화하여 효율성을 극대화합니다.
- **방안**:
  - **L1 (Low-Level)**: 프레임 단위의 입력 주입, 애니메이션 대기, 입력 안전성 검증을 담당하는 물리 계층.
  - **L2 (Mid-Level)**: 경로 탐색(A* 알고리즘), 단순 대화창 넘기기, 배틀 시 기술 선택 등 규칙 기반의 실행 계층.
  - **L3 (High-Level)**: 전체적인 스토리 목표 설정, 아이템 관리 전략, 파티 구성 등 LLM이 담당하는 전략적 의사결정 계층.

### 포켓몬 포획 이벤트 파이프라인
- **목적**: 야생 포켓몬과의 전투 시 단순히 승리하는 것을 넘어, 포켓몬을 포획하여 파티를 확장하는 시나리오를 구현합니다.
- **방안**: 배틀 진입 -> 적 포켓몬 HP 약화 -> 가방에서 몬스터볼 선택 및 투척 -> 포획 성공 여부 판정 -> 실패 시 재시도 또는 도망치기로 이어지는 상태 머신(State Machine) 기반의 포획 파이프라인을 구축합니다.

### MCP 서버 멀티 에뮬레이터 지원
- **목적**: Model Context Protocol(MCP)을 활용하여 여러 에뮬레이터 인스턴스를 동시에 제어하고 모니터링합니다.
- **방안**: 중앙 MCP 서버가 다중 에뮬레이터 게이트웨이 역할을 수행하며, 각 인스턴스의 화면과 메모리 상태를 표준화된 API로 LLM 에이전트 그룹에 제공하는 분산 제어 환경을 구축합니다.

## 3. P2: 장기 연구 및 개발 목표 (Long-Term Goals)

인간의 개입 없이 스스로 학습하고 성장하는 완전 자율형 에이전트 시스템을 지향합니다.

### 완전 자동화 코딩 에이전트
- **목적**: 에이전트가 게임을 플레이하다가 하네스 시스템의 버그나 성능 한계를 발견하면, 스스로 소스 코드를 수정하고 개선하는 자가 개선(Self-Improvement) 루프를 실현합니다.
- **방안**: 코드 분석, 테스트 실행, 패치 적용 권한을 가진 코딩 에이전트 모듈을 하네스에 통합합니다.

### 로컬 최적 대응 및 다양성 보존
- **목적**: 특정 구간에서 무한 루프에 빠지거나 동일한 행동만 반복하는 로컬 미니마 문제를 원천적으로 해결합니다.
- **방안**: 에이전트의 행동 다양성을 측정하는 메트릭을 강화하고, 탐색(Exploration) 강도를 동적으로 조절하는 알고리즘을 적용하여 새로운 경로와 전략을 지속적으로 시도하도록 유도합니다.

### 장기 메모리 시스템
- **목적**: 세션이 완전히 종료되거나 새로운 게임을 시작하더라도, 이전 플레이에서 습득한 전략적 지식과 맵 정보를 영구적으로 보존합니다.
- **방안**: 벡터 데이터베이스(Vector DB)를 활용하여 게임 내 주요 랜드마크, 효율적인 포켓몬 상성 정보, 위험 지역 목록 등을 지식 그래프 형태로 축적하고 필요할 때 검색(RAG)하여 사용합니다.

## 4. 기술 스택의 진화

| 구분 | v1 (현재) | v2 (다음) | v3 (장기) |
| :--- | :--- | :--- | :--- |
| **에뮬레이터** | mGBA | PyBoy | PyBoy + MCP |
| **제어 언어** | Lua + TypeScript | Python / TS 하이브리드 | Python / TS 하이브리드 |
| **통신 방식** | HTTP / WebSockets | 프로세스 내 직접 호출 | MCP 분산 프로토콜 |
| **의사결정** | 단일 LLM 턴 루프 | 다층 아키텍처 (L1/L2/L3) | 멀티 에이전트 협업 |
| **메모리** | 단기 세션 메모리 | 세션 핸드오프 + 마스킹 | 벡터 DB 기반 장기 메모리 |

## 5. 참조 프로젝트 및 학계 동향

- **PokeAgent Challenge (NeurIPS 2025)**: 규칙 기반 시스템과 LLM의 하이브리드 아키텍처가 순수 강화학습 에이전트보다 스토리 진행 속도와 범용성 면에서 우수함을 입증했습니다.
- **Continual Harness**: 에이전트가 게임을 완주할 때까지 멈추지 않고 스스로 하네스 코드를 수정하며 환경에 적응하는 자가 개선 프레임워크 연구입니다.
- **Metamon**: 오프라인 강화학습(Offline RL)을 통해 인간 프로게이머 수준의 포켓몬 배틀 전략을 학습한 모델로, 배틀 L2 계층 설계의 벤치마크 대상입니다.
- **PokeLoop**: 유전 알고리즘(Genetic Algorithm)을 사용하여 에이전트의 프롬프트 템플릿을 세대별로 진화시켜 최적의 탐색 효율을 찾는 프로젝트입니다.

---

## 6. 최종 비전

에이전트 시스템의 궁극적인 지향점은 다음과 같은 폐쇄형 피드백 루프(Closed Feedback Loop)의 완성입니다.

```
관찰 (Observation) ──> 판단 (Decision) ──> 실행 (Execution)
  ▲                                            │
  │                                            ▼
분석 (Analysis) <── 개선 (Improvement) <── 기록 (Logging)
```

이 루프는 실행 속도가 느리더라도 시스템 내부에서 끊임없이 회전하며 자가 진단과 코드 수정을 반복합니다. 결과적으로 인간 개발자의 개입 없이도 시간이 흐름에 따라 스스로 게임 완주율을 높여가는 지속 가능한 자율 시스템을 구축하는 것이 최종 목표입니다.
