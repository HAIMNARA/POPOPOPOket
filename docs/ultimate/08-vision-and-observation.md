# 비전 및 관측 시스템 (Vision & Observation System)

에이전트가 게임 환경을 정확하게 인지하고 의사결정을 내리기 위해서는 화면 정보와 메모리 상태를 효과적으로 가공하여 전달하는 관측 시스템이 필요합니다. 본 문서에서는 POPOPOPOket의 비전 입력 진화 과정, 메모리 마스킹 기술, Grokemon의 관측 시스템 구조, 그리고 PokemonRedExperiments의 관측 공간 설계를 상세히 분석합니다.

---

## 1. 비전 입력의 진화 (POPOPOPOket v0 ~ v3)

POPOPOPOket 프로젝트는 LLM 에이전트에게 제공하는 이미지 입력을 최적화하기 위해 네 단계에 걸쳐 비전 시스템을 개선해 왔습니다.

- **v0: 에뮬레이터 UI 전체 스크린샷**
  - 에뮬레이터 창 전체를 캡처하여 모델에 입력으로 제공했습니다. 이 방식은 게임 화면 외에 에뮬레이터 메뉴 바, 상태 표시줄 등 불필요한 영역이 포함되어 토큰 컨텍스트가 낭비되고 모델의 집중도를 떨어뜨리는 문제가 있었습니다.

- **v1: 게임 화면만 추출 (160×144)**
  - 에뮬레이터 UI를 제외하고 Game Boy의 실제 해상도인 160×144 영역만 크롭하여 제공했습니다. 불필요한 시각적 노이즈를 제거하여 효율성을 높였습니다.

- **v2: 원본 + 좌표 그리드 분리 2이미지**
  - 캐릭터의 정확한 위치 파악을 돕기 위해 화면에 좌표 그리드(Grid)를 오버레이하여 제공하기 시작했습니다. 다만, 원본 이미지에 그리드를 직접 겹쳐 그리면 텍스트나 캐릭터 스프라이트가 가려져 모델이 상황을 오인하는 부작용이 발생했습니다. 이를 해결하기 위해 원본 이미지와 그리드가 그려진 가이드 이미지를 별도의 두 장으로 분리하여 입력으로 제공했습니다.

- **v3: 원본 + 그리드 + 메모리 마스킹**
  - 그리드 이미지에 게임 메모리에서 추출한 충돌(Collision) 정보를 시각적으로 마스킹하여 결합했습니다. 통과 가능한 타일은 초록색(green), 벽이나 장애물은 빨간색(red)으로 칠해진 마스크를 제공함으로써 에이전트가 벽에 부딪히는 문제를 근본적으로 해결했습니다.

## 2. 핵심 원칙: 비전 입력 오염 금지

비전 시스템 설계에서 가장 중요한 원칙은 **"좌표 그리드를 원본 이미지에 직접 오버레이하지 않는다"**는 점입니다.

- **이유**: 원본 이미지 위에 그리드 선이나 좌표 텍스트를 직접 겹쳐서 그리면, 모델이 게임 내의 텍스트(대화창 글자)나 포켓몬 스프라이트, 아이템 아이콘 등을 제대로 인식하지 못하고 추론 성능이 급격히 저하됩니다.
- **해결책**: 원본 게임 화면은 아무런 가공도 하지 않은 순수 상태로 유지하고, 좌표 및 충돌 정보가 포함된 가이드 이미지를 별도로 생성하여 멀티모달 입력으로 함께 전달해야 합니다.

## 3. 메모리 마스킹 생성 과정

메모리 마스킹은 게임의 WRAM 데이터를 실시간으로 파싱하여 시각적 가이드로 변환하는 기술입니다.

1. **WRAM 맵 타일 데이터 파싱**
   - Game Boy 메모리에서 현재 맵의 레이아웃과 타일 배열 데이터를 읽어옵니다.
2. **타일당 충돌 속성 확인**
   - 각 타일 번호에 해당하는 충돌 속성(통과 가능 여부, 장애물 여부 등)을 조회합니다.
3. **색상 매핑**
   - 통과 가능한 타일 영역은 초록색(green)으로, 이동이 차단된 벽이나 장애물 영역은 빨간색(red)으로 칠합니다.
4. **현재 화면 영역에 매핑**
   - 캐릭터의 현재 좌표를 중심으로 카메라 뷰포트 영역에 해당하는 마스크를 크롭하여 가이드 이미지로 렌더링합니다.

- **효과**: 이 마스킹 기법을 도입한 결과, 에이전트가 좁은 1타일 통로를 통과할 때 벽에 부딪히며 헤매던 턴 수가 수십 턴에서 단 1~2 턴으로 극적으로 감소했습니다.

---

## 4. 3단계 마스킹으로의 진화

초기 v3 마스킹은 단순히 통과 가능(green)과 차단(red)의 2단계로만 구분했습니다. 그러나 이 방식은 풀숲(Grass) 타일을 처리할 때 문제를 일으켰습니다.

- **문제점**: 풀숲은 캐릭터가 통과할 수 있지만 야생 포켓몬과 조우할 확률이 존재하는 특수한 타일입니다. 이를 단순히 green으로 표시하면 에이전트가 일반 빈 공간과 구분하지 못했고, red로 표시하면 통과 가능한 길임에도 벽으로 인식하여 우회하는 문제가 발생했습니다.
- **해결책 (3단계 마스킹)**:
  - **green**: 빈 공간 (안전하게 이동 가능)
  - **yellow**: 풀숲 (통과 가능하지만 배틀 발생 확률 있음)
  - **red**: 벽 및 장애물 (이동 차단)
- 이와 같이 노란색(yellow) 카테고리를 추가함으로써 에이전트가 풀숲의 존재를 인지하고 상황에 맞는 경로 탐색을 수행할 수 있게 되었습니다.

## 5. Grokemon 관측 시스템

Grokemon 프로젝트는 LLM 에이전트에게 텍스트 기반의 정형화된 관측 데이터를 제공하기 위해 태그 섹션과 PolicyInput 구조를 사용합니다.

### 태그 섹션 구조

에이전트에게 전달되는 텍스트 프롬프트는 다음과 같은 명확한 태그 섹션들로 나뉩니다.

```
[SESSION] - 현재 모드, 페이즈, 스텝, miniState, 목표, 마지막 액션 결과
[TOOLS] - 현재 게임 모드에서 사용 가능한 도구 API 목록
[EVENTS] - 최근 세션에서 발생한 주요 이벤트 로그
[SUPERVISOR] - 상위 감독 모듈이 제공하는 어드바이저 힌트 및 가이드
[MEMORY] - 에이전트가 유지하고 있는 메모리 항목들
[HISTORY] - 최근 실행한 10개의 명령 이력
```

### PolicyInput 구성 요소

PolicyInput은 에이전트가 행동을 결정하는 데 필요한 모든 게임 상태 정보를 포함하는 객체입니다.

- **모드별 컨텍스트**: 현재 상태가 오버월드(overworld), 배틀(battle), 대화(dialog) 중 어디에 해당하는지에 대한 세부 정보입니다.
- **맵 그래프**: 현재 맵과 연결된 인접 맵들의 연결성 정보입니다.
- **현재 맵 정보**: 전체 맵 레이아웃 또는 캐릭터 주변의 마이크로 맵 정보입니다.
- **NPC 및 워프 포인트**: 맵 내에 존재하는 NPC들의 위치와 다른 맵으로 이동하는 워프 포인트 좌표입니다.
- **파티 및 적 상태**: 보유한 포켓몬들의 HP, 레벨, 기술 정보 및 전투 중인 적 포켓몬의 상태입니다.
- **인벤토리**: 소지하고 있는 아이템 목록과 수량입니다.
- **대화 상태**: 현재 화면에 표시되고 있는 대화 텍스트와 선택지 상태입니다.

## 6. PokemonRedExperiments 관측 공간

강화학습 환경인 PokemonRedExperiments는 신경망 모델의 입력으로 사용하기 위해 관측 공간을 수치화된 텐서 형태로 정의합니다. V2 Dict 관측 공간의 구성은 다음과 같습니다.

- **screens**: `(72, 80, 3)` 크기의 텐서입니다. 최근 3프레임의 화면을 다운샘플링하여 스택한 이미지 데이터로, 화면의 동적인 변화를 포착합니다.
- **health**: `(1,)` 크기의 텐서로, 현재 전투 중인 포켓몬의 HP 비율(0.0 ~ 1.0)을 나타냅니다.
- **level**: `(8,)` 크기의 텐서로, 파티 포켓몬들의 레벨 정보를 Fourier 인코딩 기법을 사용하여 표현합니다.
- **badges**: `MultiBinary(8)` 형태로, 관동 지방의 8개 체육관 배지 획득 여부를 이진 벡터로 나타냅니다.
- **events**: `MultiBinary(1024)` 형태로, 게임 내의 주요 스토리 이벤트 플래그 1024개의 활성화 여부를 나타냅니다.
- **map**: `(48, 48, 1)` 크기의 텐서로, 캐릭터 주변의 로컬 탐색 맵 상태를 표현합니다.
- **recent_actions**: `MultiDiscrete([7, 7, 7])` 형태로, 최근 실행한 3개의 액션 이력을 저장합니다.
