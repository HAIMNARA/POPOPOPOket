# 04. 에이전트 설계 패턴 (Agent Design Patterns)

포켓몬스터 레드 버전을 자동으로 플레이하기 위해 개발된 에이전트 설계 패턴을 정리합니다. 본 문서는 세 가지 주요 패러다임인 LLM 에이전트, 강화학습 에이전트, 하이브리드 에이전트의 구조와 세부 구현 패턴을 다룹니다.

## 패러다임 1: LLM 에이전트 (Grokemon)

Grokemon 프로젝트는 LLM을 의사결정의 핵심으로 삼아 게임을 플레이합니다. 이 시스템은 턴 기반 루프를 돌며 게임 상태를 분석하고 적절한 도구를 호출합니다.

### 턴 루프 패턴 (Turn Loop Pattern)

에이전트는 다음과 같은 순서로 매 턴을 실행합니다.

```
+-------------------------------------------------------+
|                     Start Turn                        |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 1. prepareTurn()                                      |
|    - 완료 상태 확인 및 자동 대화/배틀 손실 처리       |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 2. recordSessionMemoryEvents()                        |
|    - 세션 이벤트 드레인                              |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 3. buildAgentObservation()                            |
|    - 모드별 컨텍스트 (overworld/battle/dialog)        |
|    - 맵 그래프 (연결성)                              |
|    - NPC 위치 및 워프 포인트                          |
|    - 파티/적 상태, 인벤토리, 대화 상태                |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 4. session.sendUserMessage()                          |
|    - LLM 호출                                         |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 5. consumeRunEvents()                                 |
|    - 도구 호출 처리 (게임 액션 후 즉시 인터럽트)      |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 6. refreshState()                                     |
|    - 게임 상태 동기화                                 |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 7. recordTurn()                                       |
|    - 증거 기록 (JSON 데이터 및 스크린샷 저장)         |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 8. maybeAutoCheckpoint()                              |
|    - 20턴마다 자동 저장 (슬롯 8 사용)                 |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
|                      End Turn                         |
+-------------------------------------------------------+
```

턴 루프의 핵심은 게임 액션 도구를 실행한 직후 루프를 인터럽트하고 상태를 즉시 동기화하는 것입니다. 상태 갱신이 누락되면 에이전트가 이전 상태를 기준으로 잘못된 판단을 내리기 때문입니다.

### 모드별 도구 필터링 (Mode-specific Tool Filtering)

에이전트의 오작동을 방지하기 위해 현재 게임 모드에 따라 사용할 수 있는 도구를 제한합니다.

* **OVERWORLD**: navigate, interact, wait, save, load, rollback, 그리고 메모리 도구를 사용할 수 있습니다.
* **DIALOG**: dialog와 메모리 도구만 허용됩니다.
* **BATTLE**: battle, wait, 그리고 메모리 도구를 사용합니다.
* **공통 도구**: memory_read, memory_write, memory_replace, memory_delete가 항상 활성화됩니다.

이러한 필터링은 배틀 중에 필드 이동 명령을 내리거나 대화 중에 배틀 명령을 내리는 등의 비정상적인 행동을 원천 차단합니다.

### 프롬프트 구조 (Prompt Structure)

LLM에 전달되는 프롬프트는 시스템 프롬프트와 유저 메시지로 나뉩니다.

1. **시스템 프롬프트 (System Prompt)**
   * 게임 지식: 아이템 효과, 기술 상성, 포켓몬 속성, 게임 메커닉 정보를 포함합니다.
   * 모드별 규칙: overworld, battle, dialog 상황에서 지켜야 할 행동 지침을 정의합니다.

2. **유저 메시지 (User Message)**
   * 진행도: 현재 획득한 배지 개수와 누적 걸음 수를 제공합니다.
   * Supervisor 힌트: 상위 감독 시스템이 제공하는 조언을 주입합니다.
   * 마지막 명령 결과: 이전 턴에 실행한 도구의 성공 여부와 피드백을 보여줍니다.
   * 상태 정보: 캐릭터 위치, 파티 포켓몬 상태, 가방 아이템 목록, 배틀 및 대화 진행 상황을 전달합니다.
   * 맵 정보: 전체 맵 구조, 캐릭터 주변의 마이크로 컨텍스트, 연결된 맵 그래프를 포함합니다.
   * 히스토리: 최근 실행한 10개의 명령 목록을 제공하여 행동의 연속성을 유지합니다.

### 에이전트 메모리 시스템 (Agent Memory System)

메모리는 objectives, journal, notes, strategy, landmarks, lessons의 6개 섹션으로 구성됩니다. 각 섹션은 최대 20개의 항목을 가질 수 있으며, 항목당 글자 수는 500자로 제한됩니다.

메모리 데이터는 `global/agent-memory.json` 파일에 영속적으로 저장됩니다. 데이터 유실을 막기 위해 임시 파일에 먼저 쓴 뒤 이름을 바꾸는 원자적 쓰기(atomic write) 방식을 사용합니다.

### Supervisor 시스템 (Supervisor System)

에이전트의 행동을 감시하고 교착 상태를 해결하기 위해 다음과 같은 컴포넌트를 운용합니다.

* **PokemonSupervisor**: 전체 게임 진행도를 평가하고 9가지 스토리 목표 중 현재 활성화할 목표를 선택합니다.
* **StuckDetector**: 432줄 규모의 코드로 구현되어 있으며, 5가지 레벨의 막힘 현상을 감지합니다. 감지 대상은 actionLoop(4회 이상 동일 액션 반복), actionCycle, locationLoop, noProgress, backtrackLoop입니다.
* **InterventionLoop**: 막힘이 감지되면 최대 20개의 복구 입력을 실행하여 에이전트를 정상 궤도로 돌려놓습니다.
* **GoalLedger**: 현재 목표의 달성 여부를 추적하고 기록합니다.
* **LLMAdviser**: LLM을 사용하여 현재 상황을 타개하기 위한 조언을 생성합니다.

## 패러다임 2: 강화학습 에이전트 (PokemonRedExperiments)

PokemonRedExperiments는 PPO(Proximal Policy Optimization) 알고리즘을 사용하여 게임을 학습합니다. 에뮬레이터 환경을 감싸는 래퍼와 정밀한 보상 설계가 핵심입니다.

### 환경 래퍼 패턴 (Environment Wrapper Pattern)

`RedGymEnv` 클래스는 Gymnasium 인터페이스를 구현하여 에뮬레이터를 강화학습 환경으로 변환합니다.

```
+-----------------------------------------------------------------+
|                          RedGymEnv                              |
|                                                                 |
|  +-----------------------------------------------------------+  |
|  | Observation Space (Dict)                                  |  |
|  | - screens: Box(0, 255, (72, 80, 3)) [3프레임 스택]        |  |
|  | - health: Box(0, 1, (1,)) [HP 비율]                       |  |
|  | - level: Box(-1, 1, (8,)) [Fourier 인코딩]                |  |
|  | - badges: MultiBinary(8) [배지 획득 여부]                 |  |
|  | - events: MultiBinary(1024) [이벤트 플래그]               |  |
|  | - map: Box(0, 255, (48, 48, 1)) [로컬 탐색 맵]            |  |
|  | - recent_actions: MultiDiscrete([7, 7, 7]) [최근 3액션]   |  |
|  +-----------------------------------------------------------+  |
|                               |                                 |
|                               v                                 |
|  +-----------------------------------------------------------+  |
|  | Action Space: Discrete(7)                                 |  |
|  | - 방향키 4개 + A + B + Start                              |  |
|  +-----------------------------------------------------------+  |
+-----------------------------------------------------------------+
```

`step` 메서드는 에뮬레이터의 프레임 진행을 정밀하게 제어합니다.

```python
def step(self, action):
    # 프레임 0: 버튼을 누릅니다.
    # 프레임 8: 버튼을 뗍니다.
    # 프레임 24: 게임 상태를 관측합니다.
    self.pyboy.send_input(action)
    for _ in range(24):
        self.pyboy.tick()
    return obs, reward, done, truncated, info
```

### 보상 설계 (Reward Design V2)

학습의 효율성을 높이기 위해 여러 요소를 결합한 보상 함수를 사용합니다.

```python
state_scores = {
    'event': reward_scale * max_event_rew * 4,           # 이벤트 플래그 달성 (4배 가중치)
    'heal': reward_scale * total_healing_rew * 10,       # 체력 회복 행동 (10배 가중치)
    'badge': reward_scale * badges * 10,                  # 체육관 배지 획득 (10배 가중치)
    'explore': reward_scale * explore_weight * len(seen_coords) * 0.1,  # 새로운 좌표 탐색
    'stuck': reward_scale * coord_count_reward * -0.05   # 동일 좌표 정체 페널티 (600회 이상 방문 시)
}
```

이 보상 체계는 에이전트가 단순히 제자리를 맴돌지 않고 새로운 지역을 탐색하며 게임 스토리를 진행하도록 유도합니다.

### 훈련 설정 (Training Configuration)

학습 속도를 높이기 위해 다중 프로세스를 사용합니다.

```python
model = PPO("MultiInputPolicy", env,
    n_steps=2560, batch_size=512, n_epochs=1,
    gamma=0.997, ent_coef=0.01)
```

훈련은 `SubprocVecEnv`를 사용하여 64개의 병렬 에뮬레이터 환경에서 동시에 진행됩니다.

## 패러다임 3: 하이브리드 (PokeAgent / Continual Harness)

하이브리드 패러다임은 LLM의 고차원 계획 능력과 알고리즘 또는 강화학습의 실행력을 결합합니다.

### PokeAgent 아키텍처 (PokeAgent Architecture)

PokeAgent는 역할을 분담한 여러 에이전트가 협력하는 구조를 가집니다.

```
                     +--------------------------+
                     |    Orchestrator (LLM)    |
                     |       - 상위 계획        |
                     +--------------------------+
                                  |
         +------------------------+------------------------+
         |                        |                        |
         v                        v                        v
+------------------+     +------------------+     +------------------+
|Battle Strategist |     |Navigation Planner|     |Objective Verifier|
|      (LLM)       |     |   (A* + LLM)     |     |      (LLM)       |
+------------------+     +------------------+     +------------------+
         |                        |                        |
         +------------------------+------------------------+
                                  |
                                  v
                     +--------------------------+
                     |  Gym Puzzle Solver (LLM) |
                     +--------------------------+
                                  |
                                  v
                     +--------------------------+
                     |        MCP Tools         |
                     |  - A* 경로탐색           |
                     |  - 버튼 입력             |
                     |  - 지식 검색             |
                     |  - 타입 차트             |
                     +--------------------------+
```

상위 Orchestrator가 목표를 설정하면, 각 전문 에이전트가 세부 계획을 수립하고 MCP 도구를 호출하여 실행합니다.

### Continual Harness (자기개선)

에이전트가 플레이 도중 스스로 성능을 개선하는 파이프라인입니다.

```
+-------------------------------------------------------+
| Agent Rollout (256 steps)                             |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| Process Reward Model 평가                             |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 저보상 구간 감지 및 frontier teacher 재라벨링         |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| Soft SFT 업데이트 실행                                |
+-------------------------------------------------------+
                           |
                           v
+-------------------------------------------------------+
| 다음 체크포인트 저장                                  |
+-------------------------------------------------------+
```

이 루프는 시스템 프롬프트, 서브에이전트 구성, 스킬 라이브러리, 메모리를 온라인에서 동적으로 수정하며 에이전트의 행동을 교정합니다.

## 다층 아키텍처 패턴 (Multi-layer Architecture Pattern)

2026년 6월 6일 회의에서 제안된 고도화된 에이전트 설계 패턴입니다.

### 신경계 모방 3층 구조 (3-layer Nervous System Mimicry)

인간의 신경계를 모방하여 의사결정 속도와 정확도를 모두 확보합니다.

```
+-----------------------------------------------------------------+
| L3: 정책 LLM (Policy LLM)                                       |
|     - 전체 상황 감시, 목표 설정, 정책 업데이트                  |
|     - "다음에는 이러한 방향으로 행동하라" 지시                  |
+-----------------------------------------------------------------+
                               ^
                               | 보고 (Report)
+-----------------------------------------------------------------+
| L2: 비평가 (Critic)                                             |
|     - 이상 감지 및 막힘 판단                                    |
|     - "현재 상황이 비정상적인가?" 판별                          |
+-----------------------------------------------------------------+
                               ^
                               | 관찰 (Observe)
+-----------------------------------------------------------------+
| L1: 무의식 (Unconscious Layer)                                  |
|     - 결정론적 또는 휴리스틱 기반 기본 이동                     |
|     - 단순 구간 고속 통과 (fast-flow)                           |
+-----------------------------------------------------------------+
```

단순한 이동이나 반복적인 전투는 L1 레이어에서 처리하여 API 호출 비용을 줄입니다. 이상 상황이 발생하면 L2 레이어가 이를 감지하여 L3 레이어에 보고하고, L3 레이어가 새로운 정책을 수립합니다.

### 병렬 인스턴스 및 중앙 전략 가이드 (Parallel Instances & Central Strategy Guide)

여러 에이전트가 동시에 탐색하며 얻은 지식을 한곳에 모으는 구조입니다.

```
                     +--------------------------+
                     |  전략 에이전트 (상위)    |
                     |  - 웹 검색 전략 가이드   |
                     |  - VLM 화면 분석         |
                     +--------------------------+
                                  |
         +------------------------+------------------------+
         |                        |                        |
         v                        v                        v
+------------------+     +------------------+     +------------------+
|   인스턴스 1     |     |   인스턴스 2     |     |   인스턴스 N     |
+------------------+     +------------------+     +------------------+
         |                        |                        |
         +------------------------+------------------------+
                                  |
                                  v
                     +--------------------------+
                     |    중앙 전략 가이드      |
                     |  - 영속적 업데이트       |
                     |  - 최고 전략 집계        |
                     |  - 실패 아카이브         |
                     +--------------------------+
                                  ^
                                  | 견제
                     +--------------------------+
                     |  상위 웹서버 에이전트    |
                     |  - 로컬 최적화 방지      |
                     +--------------------------+
```

각 인스턴스는 독립적으로 게임을 플레이하며 성공 사례와 실패 사례를 중앙 전략 가이드에 기록합니다. 상위 웹서버 에이전트는 인스턴스들이 특정 구역에 갇히는 로컬 최적화(local optima) 현상을 방지합니다.

### 유전 알고리즘 프롬프트 진화 (PokeLoop)

프롬프트의 텍스트 구성을 유전 알고리즘을 통해 자동으로 최적화하는 실험적 패턴입니다.

1. **세대 N 실행**: 8개의 LLM 에이전트를 서로 다른 프롬프트 설정으로 병렬 실행합니다.
2. **적합도(Fitness) 평가**: RAM 데이터를 읽어 획득한 배지 수, 도감 등록 수, 맵 진행도를 기준으로 점수를 계산합니다.
3. **교차 및 돌연변이**: 우수한 성적을 거둔 프롬프트 텍스트를 결합하거나 일부 문구를 무작위로 수정합니다.
4. **세대 N+1 생성**: 새로운 프롬프트 세트를 적용하여 다음 세대 실험을 시작합니다.

이 방식을 적용한 결과, 8세대 만에 적합도 점수가 0점에서 17점으로 상승하며 첫 번째 체육관 배지를 획득하는 데 성공했습니다.

## 인간 vs AI 지속성 (Human vs AI Persistence)

자동화 에이전트와 인간 플레이어의 특징을 비교한 분석 결과입니다.

| 비교 항목 | 인간 플레이어 | AI 에이전트 |
| :--- | :--- | :--- |
| **단기 실행 속도** | 빠름 (직관적인 판단과 즉각적인 조작) | 느림 (API 호출 및 추론 대기 시간 존재) |
| **장기 지속성** | 한계 존재 (수면 및 휴식 필요) | 무제한 (24시간 연속 실행 가능) |
| **상황 관찰 방식** | 직관적인 화면 패턴 인식 | 비전 모델 및 RAM 데이터 의존 |
