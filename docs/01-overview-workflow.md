# 01 · 개요 & 워크플로우

## 1. 한 줄 정의

> "이미 실행 중인 mGBA 에뮬레이터"를 대상으로, LLM 에이전트가 매 턴 화면과 메모리를 관찰하고
> **버튼 하나**를 눌러 포켓몬 레드를 자율 플레이하는 하네스.

ROM 로딩/리셋은 모델에게 노출하지 않는다. 모델은 오직 "지금 떠 있는 게임을 진행"시키는 것만 한다.

---

## 2. 컴포넌트 구성

```
┌─────────────────────────────────────────────────────────────────┐
│  TypeScript 하네스 (에이전트 루프)                                  │
│   - 관찰: 스크린샷 + 포켓몬 RAM 상태                                  │
│   - 판단: LLM 호출 또는 결정론적 fast-flow                            │
│   - 액션: mgba_tap / mgba_hold 등 1개 실행                           │
│   - 기록: pretty-log, token-usage, run-trace, Prometheus           │
└───────────────┬─────────────────────────────────────────────────┘
                │ HTTP (127.0.0.1:5000)
┌───────────────▼─────────────────┐
│  mGBA-http (.exe)                │  HTTP API ↔ 소켓 브리지
└───────────────┬─────────────────┘
                │ TCP socket (127.0.0.1:8888)
┌───────────────▼─────────────────┐
│  mGBASocketServer.lua            │  mGBA 내부에서 구동되는 Lua 서버
└───────────────┬─────────────────┘
                │ 에뮬레이터 내부 API
┌───────────────▼─────────────────┐
│  mGBA 0.10.x + 포켓몬 레드 ROM    │  실제 게임
└──────────────────────────────────┘
```

| 컴포넌트 | 역할 | 포트 |
|---|---|---|
| mGBA | Game Boy 에뮬레이터, ROM 구동 | — |
| `mGBASocketServer.lua` | mGBA 안에서 도는 소켓 서버 (메모리 읽기/버튼 입력) | `:8888` |
| `mGBA-http` | 소켓을 HTTP API로 변환하는 브리지 | `:5000` |
| TypeScript 하네스 | 관찰→판단→액션→기록 루프 | metrics `:9464`, viewer `:9474` |

## 3. 매 턴 루프 (핵심 워크플로우)

`src/index.ts` 가 영속 세션(`pokemon-run`)을 만들고 무한 루프를 돈다. 한 턴은:

1. **관찰 수집**
   - mGBA 상태(게임 코드/타이틀/프레임), 스크린샷, 가능하면 포켓몬 RAM 상태를 캡처.
2. **스크린샷 가공**
   - Game Boy 화면을 `160×144` 로 크롭하고, 이동 가이드용 **빨간 16×16 격자**를 오버레이.
3. **입력 구성**
   - 관찰 상태 + 스크린샷 + 최근 액션 + stuck-memory 힌트를 모델 입력으로 주입.
4. **판단**
   - 결정론적으로 안전한 구간이면 **fast-flow**가 LLM 없이 액션 결정.
   - 아니면 모델이 `<action_plan>...</action_plan>` 블록 하나를 내고 **유용한 게임 액션 1개**를 실행.
5. **액션 실행 + 감독**
   - 로컬 **supervisor**가 제어 호출을 mGBA로 보내기 전에 정규화(1타일 이동 보정, 위험한 멀티홀드 거부 등).
6. **기록**
   - pretty-log, token trace, behavior metric, Prometheus 출력으로 스트리밍.

> **종료 조건 없음**: CLI 프롬프트, `--loop` 플래그, 최대 턴 제한, 완료 마커가 없다.
> 실험 창이 끝나면 `Ctrl-C` 로 프로세스를 멈춘다. (목표 마일스톤 도달 시 루프를 멈추도록 확장 가능)

## 4. 관찰 신호 (시각 + 상태)

하네스는 두 종류 신호를 결합한다:

- **시각(screenshot)**: PNG 디코드 → Game Boy 영역 크롭 → 이동 격자 오버레이 → 흑/로딩 프레임 감지.
- **상태(RAM)**: 맵 ID, 좌표(x,y), 바라보는 방향, 전투 상태 등 컴팩트 필드 읽기.
  RAM 읽기에 실패하면 **시각 전용 관찰로 폴백**한다.

상세 주소는 [04 · RAM 맵 & 조작 레퍼런스](04-ram-and-controls.md) 참고.

## 6. 메트릭 & 트레이스

각 실행은 `.pss-mgba/traces/runs/<run-id>/` 아래에 트레이스 디렉토리를 만든다.

| 출력 | 내용 |
|---|---|
| `run.json` | 실행 메타데이터, 모드, experiment id, 마일스톤, 막힘 수, 슈퍼바이저 개입 수 |
| `events.jsonl` | 관찰 스크린샷/상태, `action_plan` 요약, 액션 툴 호출/결과, 슈퍼바이저 개입 로그 |
| `token-usage.jsonl` | 스텝별/턴별 토큰 사용량 |
| Prometheus | 기본 `http://127.0.0.1:9464/metrics` |

행동 메트릭에는 액션 엔트로피, A버튼 비율, 동일 액션 연속, 시각 신규성,
"행동 전 관찰" 비율, 툴 에러율, 턴/스텝/툴 소요시간, 막힘 이벤트, 슈퍼바이저 개입이 포함된다.

> **토큰 절감은 "진행·막힘·액션 다양성·툴 신뢰성이 퇴보하지 않을 때만" 개선으로 인정**한다.

---

## 7. 실행 모드

유효 모드: `fresh`, `resumed`, `recovery`, `deterministic-replay`, `exploratory`.

- `fresh`: 현재 라이브 에뮬레이터 상태에서의 정상 실행. 진행 메트릭 비교는 이 모드끼리만.
- `recovery` / `deterministic-replay` 메트릭은 `fresh` 진행 메트릭과 **섞지 않는다**.
