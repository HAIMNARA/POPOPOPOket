# 07 · 함정 아카이브 — 하드원(hard-won) 교훈

실제로 막히고 깨지면서 알아낸 것들. **재발 방지**가 목적. 같은 함정에 다시 빠지지 말 것.

---

## 🚨 1. mGBA `--script` CLI 옵션은 존재하지 않는다 (키스톤 블로커)

- **증상**: `mgba --script <lua> <rom>` 실행 시 `unknown option -- script` 에러 후 **즉시 종료**.
  mgba-sdl도, `config.ini`도 스크립트 자동로드를 지원하지 않음.
- **결과**: 이 사실을 모르면 런처가 mGBA를 띄우자마자 죽어서 **하네스 자체가 100% 구동 불가**였다.
  (stderr 캡처로 직접 증명함.)
- **해결**:
  - 런처(`start-harness.ps1`)는 **mGBA + ROM만 실행**(창은 살아있음)하고,
    Lua는 **GUI로 한 번 로드**하라고 ACTION-REQUIRED 안내를 출력.
  - `:5000` 폴링을 180초로 연장.
- **Lua는 반드시 GUI 로드**: `Tools → Scripting → File → Load script`.
  CLI로는 불가능하므로 이 한 단계만 사람이 해준다.

---

## 🚨 2. Lua 소켓 서버를 두 번 로드하거나 `dofile`로 로드하면 mGBA 크래시

- **금지 1**: 스크립팅 콘솔에서 `dofile([[...]])` 로 로드.
- **금지 2**: 스크립트를 **2회 이상** 로드.
- **이유**: 둘 다 관리되지 않는 소켓/프레임 콜백을 만들어 **use-after-free 크래시**.
- **정상 신호**: 콘솔에 `mGBA script server 0.8.2 ready. Listening on port 8888`.
- **진단 팁**: `:5000`이 200을 안 주면 `:8888`(Lua 소켓) 리슨 여부부터 확인.
  - `:8888` 닫힘 → Lua가 바인딩 안 됨(미로드/크래시) → mGBA 재실행 후 GUI로 로드.
  - `:8888` 열림 + `:5000` 실패 → mGBA-http만 재시작.

---

## 🚨 3. Lua 로드 GUI 자동화는 신뢰성이 낮다

- 스크립팅 창 + **한글 경로 파일 다이얼로그** 멀티윈도우 조작은 너무 불안정.
- 여러 자동화 시도(`loadlua*.ps1`, `loadgui.ps1`)가 모두 깨졌음.
- **결론**: 자동화하려 애쓰지 말고 **"한 번만 수동 GUI 로드"** 를 문서화된 단계로 남긴다.

---

## ⚠️ 4. 토크나이저/모델 토큰 소모 — "토큰 닳는 속도" 측정의 함정

같은 작업이라도 "토큰 닳는다"의 의미가 3가지로 갈린다. 비교할 땐 단위를 명시할 것:

| 측정 단위 | 의미 |
|---|---|
| API 청구 토큰 | 실제 비용. 토크나이저/verbosity 영향 |
| 사용량 게이지(구독 quota) | 모델 + 툴 콜 + thinking 종합 |
| 작업 완수까지 누적 비용 | 재시도/자가수정 포함 |

- **교훈**: "체감 빨리 닳는다"와 "실제 청구서"는 다르다. effort 셋팅·재시도 횟수를 같이 봐야 한다.
- **출처 위생**: 모델 스펙 같은 사실 주장은 1차 자료(공식 문서)로 grounding. SEO/AI생성 의심 사이트와
  커뮤니티 일화는 분위기 파악용일 뿐 사실 근거로 인용 금지.

## ⚠️ 5. 마일스톤 회귀 — Viridian = mapId 1 충돌

- Viridian City가 `mapId = 0x01` 이라, 기존 "first map transition" 테스트가 일반 다른 맵으로
  `mapId:1` 을 쓰면 Viridian 분기와 **충돌**한다.
- **해결**: 그 테스트를 `mapId:2` 로 옮겨 의도(맵 변경 → 첫 전환)를 보존하고,
  `mapId:1` 은 Viridian 전용으로 예약. Viridian 체크를 null 가드 직후 **최우선 양성 마일스톤**으로 배치.
- 기존 9개 테스트 케이스(대부분 mapId 0/null) 대비 안전함을 검증.

---

## ⚠️ 6. 대사/메뉴 감지는 RAM만으로 안 된다

- `dialogueLike` / `menuLike` 는 RAM 신뢰 판정이 어려워 기본값이 `"visual-fallback"`.
- 그래서 대사 auto-A, 이름입력 auto-START 자동화(4a/4b)는 **프레임버퍼 휴리스틱(시각 보정)** 이 전제.
  이게 라이브 보정의 **#1 리스크**였다.
- fast-flow 결정 로직은 작성·테스트됐지만, 과거 시점엔 `index.ts`에 완전 와이어링되지 않았고
  대사/메뉴 분기는 항상 `visual-fallback`이라 **죽은 가지**였다.

---

## ⚠️ 7. fast-flow 우회는 `index.ts` 턴 루프에 넣어라 (supervisor 아님)

- `supervisor.ts` 는 타이밍만 다루는 HTTP 프록시.
- LLM 우회(자동조종) 판단은 **`index.ts` 의 while 루프, `session.send` 직전**에 들어가야 한다.
- 슈퍼바이저에 우회 로직을 넣으면 안 된다(책임 분리).

---

## ⚠️ 8. 증거(evidence)는 run id + ROM identity에 묶어라

- 토큰 사용량만 좋아지고 진행/막힘/액션 엔트로피/툴 신뢰성/ROM identity가 나빠지면 **롤백**.
- **ROM identity 혼동 주의**: 한 베이스라인은 포켓몬 레드(`DMG-AR` / `PKMN RED ST`),
  다른 실행은 포켓몬 골드(`DMG-AAUE` / `POKEMON_GLD`)였다.
  서로 다른 ROM의 메트릭을 섞어 "레드 개선"이라 주장하면 안 된다.
- `recovery` / `deterministic-replay` 메트릭을 `fresh` 진행 메트릭과 섞지 말 것.

---

## ⚠️ 9. 라이브 실행 = 단일 에뮬레이터 원칙

- 라이브 실험은 **하나의 에뮬레이터 서버**만 띄운다.
- 두 번째 mGBA나 mGBA-http를 띄우지 말 것 — 현재 에뮬레이터 상태가 곧 측정 대상이다.
- 고아 `mGBA-http` 프로세스는 정리(kill)할 것.

---

## ✅ 빠른 진단 체크리스트

| 증상 | 1순위 확인 |
|---|---|
| mGBA가 켜자마자 꺼짐 | `--script` 옵션 썼는지 (쓰지 말 것) |
| `:5000` 200 안 줌 | `:8888` 리슨 여부 → Lua GUI 로드 확인 |
| mGBA 크래시 | Lua 2회 로드 / `dofile` 사용 여부 |
| RAM 상태 unavailable | mGBA-http 연결 + WRAM 주소 → 시각 폴백 동작 확인 |
| 토큰 폭주 | effort 셋팅, fast-flow 와이어링, 막힘 반복 여부 |
| 진행은 그대론데 토큰만 줄음 | 개선 아님 — 롤백 후보 |
