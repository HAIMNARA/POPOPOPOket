## 1. 세션 요약

포켓몬 레드 AI 에이전트 자동화 프로젝트의 발표 및 피어 리뷰 세션.
참가자들이 각자 구현한 하네스 시스템을 시연하고,
에이전트의 게임 조작 방식 · 메모리 활용 · 다층 구조 설계 등을 논의하였다.

## 2. 기본 하네스 루프 (공통 기반)

모든 참가자가 공유한 최소 루프:

```
┌─ 에뮬레이터 스크린샷 캡처 ─────────────────────────┐
│                                                     │
▼                                                     │
모델에 이미지 입력                                      │
  → 버튼 배열 출력 (function call)                      │
  → 에뮬레이터에 프로그래매틱 입력                        │
  → 결과 스크린샷 피드백 ──────────────────────────────┘
```

- **도구 스키마**: Anthropic 툴 스키마 형식, `use_emulator` 함수
  - 입력: 버튼 스트링 배열 (`["UP", "A", "B", ...]`)
  - 추가 파라미터: `WAIT` 타이머 (에이전트가 대기 후 응답 수신)
- **조작 가능 버튼**: `UP`, `DOWN`, `LEFT`, `RIGHT`, `A`, `B`, `START`, `SELECT`
  - 포켓몬 레드는 게임보이(구세대) 전용 → L/R 버튼 없음

## 3. 하네스 개선 히스토리

### 3-1. 이미지 크로핑

| 단계 | 방식 | 문제점 |
|------|------|--------|
| 초기 | 에뮬레이터 UI 프레임 전체 입력 | 게임 화면 외 영역이 컨텍스트 낭비 |
| 개선 | 게임 화면만 크로핑(160x144) | 해결 — 진행에 필요한 정보만 전달 |

### 3-2. 좌표계 시스템

| 단계 | 방식 | 문제점 |
|------|------|--------|
| 초기 | 게임 화면에 빨간 16x16 그리드 오버레이 | **이미지 오염** — 원본 시각 정보 손실, 모델 추론 근거 부족 |
| 개선 | **원본 이미지 + 좌표계 이미지를 별도 2장** 제공 | 해결 — 원본 보존 + 격자 기반 인식 동시 확보 |

### 3-3. 메모리 맵 기반 마스킹 (핵심 개선)

게임보이 WRAM 메모리를 파싱하여 맵 데이터를 추출하고,
이동 가능 영역(초록) / 차단 영역(빨간)을 시각적으로 구분한 **마스킹 이미지**를 추가 입력으로 제공.

**효과**:
- **마스킹 전**: 에이전트가 벽을 무작위로 탐색하다 우연히 통과
- **마스킹 후**: 개구부를 즉시 인식하고 직행

특히 **육안으로도 식별 어려운 좁은 통로 구간**에서 극적인 개선:
- 벽으로 둘러싸인 곳에 사람 눈으로도 안 보이는 1타일 개구부가 있는 경우
- 마스킹 없이는 모델이 수십 번 벽에 부딪히다 운 좋게 통과
- 마스킹 적용 후 초록색 개구부를 보고 즉시 직행

### 3-4. 메타 레이어 & 정책 업데이트

단순 비전 태스크의 한계를 인식하고, 다음 구조로 발전:

1. **이동 시도 기록**: 매 이동마다 갈 수 있는 곳 / 막힌 곳을 지속 기록
2. **메타 레이어**: 기록된 데이터를 판단하는 상위 레이어
3. **정책 업데이트**: 에이전트가 스스로 경로를 판단하고 정책을 갱신

> "닫힌 피드백 루프(closed loop)를 구성하면 속도가 느리더라도 지속적 개선이 발생한다"
> — 이번 이벤트에서 검증된 핵심 가설. 모든 도메인에 적용 가능.

## 4. 참가자별 하네스 아키텍처

### 4-A. 다층 신경계 모사 구조

인간 신경계를 모사한 3계층 설계:

```
┌─────────────────────────────────────────────────┐
│  [L3] 상위 LLM — 정책 생성/조정, 전체 감시       │
│       "나 양치질 해야지" 수준의 의사결정           │
├─────────────────────────────────────────────────┤
│  [L2] 자율신경계 크리틱 — 이상 감지               │
│       "지금 상황이 이상한가?" 판별                │
├─────────────────────────────────────────────────┤
│  [L1] 무의식적 기본 이동 — 생각 없이 하는 행동    │
│       결정론적/휴리스틱 이동                      │
└─────────────────────────────────────────────────┘
```

- L1: 아무 생각 없이 자연스럽게 이동하는 단계
- L2: 이동의 이상 여부를 감지하는 크리틱 프로세스
- L3: 두 하위 레이어를 감시하고 정책을 생성/조정하는 LLM
- **장기 메모리** 추가 시 더 고도화 가능
- **코딩 태스크 등 다른 도메인에도 적용 가능**한 범용 구조

### 4-B. 오목 에이전트 기반 병렬 탐색 구조

오목 에이전트에서 검증된 구조를 포켓몬 하네스에 적용:

```
┌─ 메인 리즈닝 모델 (추론/의사결정) ──┐
│                                     │
├─ 서브 에이전트 A (병렬 경로 탐색) ──┤
├─ 서브 에이전트 B (병렬 경로 탐색) ──┤
├─ 서브 에이전트 C (병렬 경로 탐색) ──┤
│                                     │
│  → 탐색 결과 종합 → 행동 결정       │
└─────────────────────────────────────┘
```

- 서브 에이전트들이 병렬로 경로를 탐색
- 상위 리즈닝 모델이 결과를 종합하여 행동 결정
- **이슈**: Lazy Codex 환경에서 콜 호스트가 주기적으로 끊기는 안정성 문제 발생

### 4-C. 병렬 인스턴스 + 중앙화된 공략집

```
┌─ 전략 수립 에이전트 (상위) ─────────────────────┐
│  웹 서칭으로 시드 공략집 확보                      │
│  VLM이 화면 분석 → 서브 목표 생성                 │
├────────────────────────────────────────────────┤
│                                                 │
│  ┌─ 하네스 인스턴스 1 ─┐  ┌─ 인스턴스 2 ─┐  …  │
│  │ 마이크로 액션 수행   │  │              │      │
│  │ 실수/개선 → 보고    │  │              │      │
│  └──────┬──────────────┘  └──────┬───────┘      │
│         │                        │               │
│         ▼                        ▼               │
│  ┌─────────────────────────────────────────┐    │
│  │      중앙화된 공략집 (지속 업데이트)       │    │
│  │  베스트 전략 집약 / 실수 사례 아카이브     │    │
│  └─────────────────────────────────────────┘    │
├────────────────────────────────────────────────┤
│  상위 웹 서버 에이전트 — local optima 견제       │
│  (시간 부족으로 완전 검증 미완)                   │
└────────────────────────────────────────────────┘
```

- 전략 에이전트와 마이크로 액션 에이전트를 분리한 계층 구조
- 여러 하네스 인스턴스를 병렬 실행
- 각 인스턴스의 실수/개선 사례를 **중앙화된 공략집에 지속 업데이트**
- **MCP 서버** 활용 시 에뮬레이터 다중 실행 가능 (뒤늦게 공유됨)
- 연합 강화 학습 프레임워크와 유사 — **local optima 견제** 필요

### 4-D. 논문 기반 접근 + 저비용 모델 운용

- 포켓몬 관련 논문(딥마인드 발표 포함)을 LLM으로 먼저 분석 → 공략 지식 사전 주입
- **Genspark** 플랫폼 + **GPT-5.4** 모델로 저비용 운용 실험
- 초기 단계(이름 입력까지)는 빠르게 진행되나, 맵 탈출 구간에서 동일한 매핑 이슈에 직면

## 5. 핵심 인사이트

### 5-1. 인간 vs. AI 에이전트 지속성

| 관점 | 인간 | AI 에이전트 |
|------|------|-------------|
| 단기 속도 | **우위** | 느림 |
| 장기 지속성 | 수면 등 생리적 한계 | **역전** |

실제 이벤트 중 인간이 먼저 풀고 있었으나, 수면 중 AI 에이전트에 역전당한 사례 발생.
→ "바이오 토큰(인간 인지 자원)" vs "GPT 토큰(API 비용)"의 트레이드오프.

### 5-2. 하네스 엔지니어링의 본질

> 인간이 게임을 플레이하는 관점에서 에이전트의 행동을 관찰하고,
> 부족한 부분을 시스템으로 보완하는 **반복적 개선 과정**이 핵심.

이 접근법은 포켓몬 외 **모든 도메인에 적용 가능한 범용 원칙**으로 제시되었다.

### 5-3. 닫힌 피드백 루프 가설

> "티켓/이슈 시스템을 통해 닫힌 루프(closed loop)만 구성하면,
> 속도가 느리더라도 지속적인 개선이 발생한다."

이번 이벤트를 통해 **검증**되었으며, 다른 도메인에 확장 적용할 수 있다는 결론 도출.

### 5-4. 풀숲 오인식 문제

에이전트가 통과 가능한 풀숲을 벽으로 인식하여 **시도조차 하지 않는** 문제가 다수 보고됨.
- 마스킹 시스템으로 부분 해결 가능
- 완전한 해결을 위해서는 풀숲/잔디 타일에 대한 추가 분류 필요

### 5-5. 컨텍스트 포화 & 세션 핸드오프

- 장시간 실행 시 **컨텍스트 메모리가 꽉 차서** 새 세션으로 전환해야 하는 상황 빈발
- 핸드오프 시 **MD 파일로 이력을 정규화**하여 다음 세션에 효율적으로 전달하는 방안 논의
- **컴팩션(compaction)** 및 **프루닝(pruning)** 전략이 핵심 과제

### 5-6. 초기 맵 탈출 문제 (침실 스폰)

에이전트가 침실(2F)에 스폰되면 나가는 출구(계단)가 화면에 보이지 않음.
- 위로 이동해야 계단이 보이는데, 에이전트는 "바닥을 기면서 계단을 찾는" 행동 반복
- 공략집도 "계단 찾아 나가라"로만 기술되어 있어 LLM에게도 부족한 정보
- → 초기 스폰 지점에서의 가이드 프롬프트가 필요
