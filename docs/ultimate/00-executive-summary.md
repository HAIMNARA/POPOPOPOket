# 포켓몬스터 레드 AI 자동화 프로젝트 종합 개요

본 보고서는 포켓몬스터 레드 버전을 자동으로 플레이하기 위해 시도된 세 가지 주요 프로젝트의 접근법, 아키텍처, 성과를 분석하고 향후 개발을 위한 최적의 방향을 제시합니다.

## 3개 프로젝트 소개

### 1. Grokemon (INONONO66/grokemon)

Grokemon 프로젝트는 Grok 4.3 LLM 에이전트를 기반으로 포켓몬스터 레드를 자동 플레이하는 시스템입니다. 개발진은 TypeScript 언어를 사용해 약 200개의 커밋으로 코드를 구축했으며, pnpm workspace 환경에서 관리합니다. 전체 아키텍처는 CommandAgentRunner, CommandExecutor, InputGate, mGBA-http 모듈로 이어지는 구조를 가집니다.

에이전트는 관측, LLM 호출, 도구 실행, 상태 갱신으로 순환하는 턴 기반 루프를 수행합니다. 게임 상황에 맞춰 overworld, battle, dialog 모드별로 도구를 필터링하여 오작동을 방지합니다. wJoyIgnore 및 wWalkCounter 메모리 주소를 감시하는 InputGate 덕분에 입력 안전성을 확보했습니다.

메모리 구조는 objectives, journal, notes, strategy, landmarks, lessons의 6개 섹션으로 나뉩니다. 길 찾기에는 A* 경로탐색 알고리즘을 적용했고 NPC 장애물 감지 기능도 포함했습니다. 목표 원장과 막힘 감지 기능을 갖춘 Supervisor가 에이전트의 행동을 감독하며 개입 루프를 돌립니다. 모든 턴의 실행 결과는 JSON 데이터와 스크린샷으로 기록되어 증거로 남습니다. navigate, interact, wait, dialog, battle의 5가지 게임 액션 도구를 제공합니다. 에뮬레이터 환경은 mGBA에 HTTP API를 제공하는 mGBA-http와 mGBASocketServer.lua 스크립트를 결합해 구성했습니다. 이 시스템은 약 7시간 만에 주인공의 침실에서 Viridian City까지 도달하는 성과를 냈습니다.

### 2. POPOPOPOket (HAIMNARA/POPOPOPOket)

POPOPOPOket 프로젝트는 LLM 기반 포켓몬스터 레드 자동화 과정에서 얻은 워크플로우와 공략 지식을 정리한 아카이브입니다. PowerShell 스크립트와 Markdown 문서로 구성된 이 프로젝트는 아키텍처, 환경설정, RAM 맵, 자동조종, 함정 분석 등 10개의 체계적인 문서를 포함합니다.

연구를 통해 몇 가지 핵심적인 발견을 도출했습니다. 통과 가능한 구역과 벽을 구분하는 메모리 마스킹 기법을 도입하여 기존에 수십 턴이 걸리던 탐색 과정을 1에서 2턴으로 단축했습니다. 에이전트가 특정 좌표에서 반복해서 실패할 때 해당 좌표 정보를 프롬프트에 주입하는 stuck-memory 기법으로 무한 루프를 방지합니다. Route 1처럼 단순하고 결정론적인 구간은 LLM 호출 없이 자동으로 통과하는 fast-flow 방식을 적용해 토큰 비용을 절약했습니다.

이슈와 티켓을 통해 지속적으로 시스템을 보완하는 닫힌 피드백 루프를 구축했고, 이는 2026년 6월 6일 회의에서 검증되었습니다. 비전 입력의 오염을 막기 위해 원본 화면 이미지와 가이드 이미지를 분리하여 제공하는 원칙을 세웠습니다. 여러 모델 중 Grok 4.3이 비전 정확도, 게임 컨텍스트 이해도, 토큰 사용량 면에서 가장 뛰어난 성능을 보였습니다. 에뮬레이터 실행 시 `--script` 옵션 사용 금지나 Lua 스크립트 중복 로드로 인한 크래시 등 15건의 함정 사례를 아카이브에 기록했습니다. 증거 없는 개선은 없으며 닫힌 루프가 존재할 때만 실질적인 개선이 일어난다는 설계 원칙을 고수합니다.

### 3. PokemonRedExperiments (PWhiddy/PokemonRedExperiments)

PokemonRedExperiments 프로젝트는 Stable Baselines 3의 PPO 알고리즘과 PyBoy 에뮬레이터를 결합한 강화학습 기반 자동화 시도입니다. Python 언어로 작성되었으며 GitHub에서 7.9k 이상의 Star와 272개의 커밋을 기록하며 큰 관심을 받았습니다. 아키텍처는 PPO Policy가 Gym wrapper인 RedGymEnv를 거쳐 PyBoy 에뮬레이터를 제어하는 구조입니다.

초기 버전인 V1에서는 4320차원 프레임 벡터를 기반으로 한 KNN 탐색 보상을 사용했습니다. 후속 버전인 V2에서는 x, y 좌표와 맵 ID를 조합한 튜플 카운팅 방식을 도입하여 훈련 속도를 4배 향상시켰습니다. 보상 체계는 이벤트 플래그, 체육관 배지 획득, 체력 회복, 탐색 진행도, 막힘 감지 패널티 등으로 세밀하게 구성됩니다. 관측 공간은 화면 이미지, 체력, 레벨, 배지, 이벤트 상태, 현재 맵 정보, 최근 행동 기록을 포함하는 딕셔너리 형태를 취합니다.

훈련 속도를 높이기 위해 SubprocVecEnv를 사용해 64개의 병렬 환경을 동시에 구동합니다. 학습 과정의 지표는 TensorBoard와 Weights & Biases를 통해 실시간으로 로깅됩니다. WebSocket StreamWrapper를 구현하여 라이브 방송 중에 여러 에이전트가 탐색한 영역을 통합한 공유 글로벌 맵을 시청자에게 보여줍니다. 이 강화학습 모델은 수많은 반복 학습 끝에 Cerulean City까지 도달하는 데 성공했습니다. PyBoy 에뮬레이터는 Python 네이티브 환경에서 단일 패키지로 동작하여 연산 속도가 매우 빠릅니다. 연구 성과는 "Pokemon Red via Reinforcement Learning"이라는 제목의 논문(arXiv 2502.19920)으로 발표되었습니다.

## 광범위한 생태계 동향 (2025년에서 2026년)

최근 포켓몬스터 AI 자동화 분야에서 주목할 만한 프로젝트들이 다수 등장했습니다. NeurIPS 2025에서 개최된 PokeAgent Challenge는 배틀과 스피드런의 두 가지 트랙으로 진행되었습니다. 이 대회에서는 LLM이 목표를 분해하고 RL이 세부 행동을 실행하는 하이브리드 접근법이 가장 우수한 성과를 거두었습니다. Continual Harness 프로젝트는 자기개선 능력을 갖춘 LLM 에이전트를 설계하여 Gemini 3 Pro 모델로 포켓몬 블루, 옐로, 크리스탈 버전을 최초로 완주하는 기록을 세웠습니다. Metamon은 47만 5천 건 이상의 대전 리플레이 데이터를 학습한 오프라인 RL 모델로 포켓몬 쇼다운에서 인간 수준의 실력을 보여주었습니다. PokéAI는 계획, 실행, 비평 루프를 반복하는 텍스트 기반 멀티에이전트 아키텍처를 제안했습니다. PokeLoop 프로젝트는 유전 알고리즘을 적용해 LLM 프롬프트를 자동으로 진화시키는 실험을 진행했습니다.

이러한 연구들을 통해 LLM과 RL의 관계에 대한 핵심적인 사실들이 밝혀졌습니다. BALROG 벤치마크 결과에 따르면 LLM은 보상 모델로서 훌륭한 역할을 수행하지만 직접 게임을 조종하는 에이전트로서는 한계를 보였습니다. 상위 계획은 LLM이 담당하고 하위 실행은 RL이 처리하는 하이브리드 구조가 순수 LLM이나 순수 RL 단독 모델보다 뛰어난 성능을 냅니다. 비전 모델의 경우 텍스트 정보만 제공할 때보다 이미지를 함께 입력할 때 오히려 성능이 떨어지는 비전 역설 현상이 관찰되었습니다. 장기적인 관점에서는 인간의 일시적인 집중력보다 AI의 24시간 지속적인 탐색 능력이 게임 클리어 속도 면에서 더 유리하다는 점이 입증되었습니다.

## 최적 접근법 요약

목표 기간에 따라 다음과 같은 단계별 접근법을 권장합니다.

1. **단기 목표 (Viridian City 도달)**: Grokemon 스타일의 LLM 에이전트 구조에 메모리 마스킹과 stuck-memory 기법을 결합하는 방식이 적합합니다. 이 조합은 탐색 효율을 극대화하고 무한 루프 문제를 효과적으로 해결합니다.
2. **중기 목표 (체육관 배지 획득)**: LLM이 상위 계획을 수립하고 결정론적인 하위 스크립트나 알고리즘이 실행을 담당하는 하이브리드 모델을 권장합니다. 복잡한 전투나 길 찾기 상황에서 토큰 소모를 줄이면서 안정성을 높일 수 있습니다.
3. **장기 목표 (게임 전체 클리어)**: L1 무의식 레이어, L2 비평가 레이어, L3 정책 LLM 레이어로 구성된 다층 아키텍처를 도입해야 합니다. 이와 함께 중앙 전략 가이드를 자동으로 업데이트하는 시스템을 구축하는 것이 필수적입니다.

## 핵심 발견 및 권장사항

첫째, 메모리 마스킹과 stuck-memory 기법은 LLM 에이전트의 탐색 속도를 높이는 핵심 요소입니다. 벽과 통로를 구분하는 마스킹을 통해 불필요한 행동을 줄이고, 실패한 좌표 정보를 프롬프트에 주입하여 막힘 현상을 방지해야 합니다.

둘째, 단순 이동 구간에서는 LLM 호출을 생략하는 fast-flow 설계를 도입해야 합니다. Route 1과 같이 단순한 경로는 결정론적 코드로 처리하여 불필요한 API 비용을 절감하는 것이 바람직합니다.

셋째, 비전 모델 입력 시 원본 이미지와 가이드 이미지를 철저히 분리해야 합니다. 화면 정보가 오염되면 모델의 판단력이 저하되므로 입력 데이터를 정제하여 제공하는 규칙을 준수해야 합니다.

넷째, 에뮬레이터 구동 환경의 안정성을 확보해야 합니다. Lua 스크립트의 중복 로드나 잘못된 실행 옵션은 시스템 크래시를 유발하므로 예외 처리와 환경 검증을 철저히 수행해야 합니다.

다섯째, 상위 계획과 하위 실행을 분리하는 하이브리드 아키텍처를 지향해야 합니다. LLM의 전략 수립 능력과 강화학습 또는 스크립트의 실행력을 결합할 때 가장 높은 성공률을 기대할 수 있습니다.
